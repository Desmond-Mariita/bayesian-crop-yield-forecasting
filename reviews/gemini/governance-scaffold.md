# Code Review: Initial Project Setup and XAI Contract

## Verdict: APPROVE-WITH-CHANGES

The project setup is exceptionally thorough, with a clear charter, a well-structured curriculum, and an impressive governance-as-code foundation. The use of CI-enforced invariants, stub contracts for TDD, and detailed documentation sets a high standard for a learning repository. The changes are approved pending a few fixes to address minor architectural inconsistencies and documentation mismatches that would undermine the project's otherwise outstanding rigor.

## Findings

1.  **Severity:** Major
    **File:** `src/xai/cards.py`, line 91
    **Clause:** `technical_detail: Mapping[str, Any] = field(default_factory=dict)`
    **Finding:** The `ExplanationCard` dataclass is declared as `frozen=True`, but its `technical_detail` field uses a mutable `dict` as a default. This creates a mutability leak, allowing consumers of a card to modify its contents after creation, violating the immutability contract. The unit test `test_card_is_immutable` only checks direct attribute assignment and misses this mutation vector.
    **Recommendation:** To enforce true immutability, use a `types.MappingProxyType` to wrap the dictionary.

    ```python
    # In src/xai/cards.py
    import types
    from dataclasses import field, dataclass

    # ... in ExplanationCard ...
    technical_detail: Mapping[str, Any] = field(default_factory=dict)

    def __post_init__(self) -> None:
        # ... existing checks ...
        object.__setattr__(self, "technical_detail", types.MappingProxyType(self.technical_detail))
    ```

2.  **Severity:** Major
    **File:** `src/xai/gates.py`, lines 107-114
    **Clause:** `evaluate_convergence` implementation logic.
    **Finding:** The documented convergence criteria (in `docs/INVARIANTS.md` LINV-003 and the function's own docstring) are "R-hat < 1.01, ESS > 400". The implementation, however, uses non-strict inequalities: `r_hat > 1.01` is a failure (allowing `r_hat == 1.01`) and `ess_bulk < 400` is a failure (allowing `ess_bulk == 400`). This discrepancy, while arguably pragmatic, violates the project's principle of rigorous contracts.
    **Recommendation:** Align the implementation and documentation. Either change the docstrings and invariants to `R-hat <= 1.01` and `ESS >= 400`, or change the operators in `evaluate_convergence` to be strict (`>= R_HAT_MAX` for r_hat, `<= ESS_BULK_MIN` for ess). The former is recommended for practicality. Also, add unit tests for the boundary conditions (e.g., `r_hat=1.01`).

3.  **Severity:** Minor
    **File:** `.github/workflows/ci.yml`, lines 52-56
    **Clause:** `pytest + coverage gate` step.
    **Finding:** The CI workflow duplicates the logic for deferring the pytest coverage gate from `tools/verify.py`. The `verify.py` script is the designated single source of truth for verification, and the CI should execute it directly rather than re-implementing its steps. This divergence creates a maintenance burden and a risk of the two systems behaving differently.
    **Recommendation:** Replace the individual check steps in `ci.yml` (guidelines, data cards, notebooks, flake8, black, mypy, pytest) with a single call to `python tools/verify.py`. The script's exit code and report will provide the necessary PASS/FAIL signal.

4.  **Severity:** Minor
    **File:** `docs/INVARIANTS.md`
    **Clause:** Invariants LINV-004 and LINV-006.
    **Finding:** The "Enforcement" for LINV-004 ("Missing data is a model input") and LINV-006 ("Wide priors") is listed as "review" or "preprocessing tests". Unlike other invariants which are enforced by specific tools (`check_data_cards.py`, `check_notebooks.py`), these are largely aspirational and rely on manual oversight. This weakens the "governance by CI" claim.
    **Recommendation:** Acknowledge this limitation directly in the `INVARIANTS.md` document. For LINV-004, a custom linting rule could be written in the future. For LINV-006, it will always remain somewhat subjective, but noting it as "Enforced by manual review of model cards" is more transparent than implying full automation.

## Explicit Checks

**1. `src/xai` correctness:**
- **Card validation:** Semantics are mostly correct. Field validation (`_require_non_empty`, `_require_confidence_in_range`) is sound. However, a major mutability leak exists in `ExplanationCard` via the `technical_detail` field's default `dict` factory (see Finding #1).
- **`EvidenceGate`:** Semantics are correct. `>`=` is correctly implemented for thresholds, and absent keys are correctly handled as zero.
- **`evaluate_convergence`:** There is a major discrepancy between the documented strict inequalities (`< 1.01`, `> 400`) and the implemented non-strict ones (`<= 1.01`, `>= 400`). See Finding #2.
- **ABC contract:** The `ExplainableModel` abstract base class is sound and correctly defines the `explain_or_reject` contract.

**2. Test quality:**
- **Behavior pinning:** The unit tests in `test_xai_cards.py` and `test_xai_gates.py` are generally good but primarily mirror the implementation. They successfully pin the implemented behavior but fail to catch the contract discrepancy in `evaluate_convergence` because they don't test against the separate documentation artifact.
- **Stub-contract engine:** The dynamic discovery in `test_stub_contracts.py` is sound and robust. Using `ast` to read the `IMPLEMENTED` ledger is a clever, safe way to avoid import side effects. The TDD-nudge workflow it creates is a major strength of the repo's design. It cannot silently mis-discover as long as new modules are added to the `STUB_MODULES` tuple, and the `len(CASES)` check provides a good guardrail.
- **`NotImplementedError` for coverage:** This is a pragmatic and effective, if slightly unconventional, strategy. It allows the project to maintain a high coverage standard from the beginning while correctly testing the contract that "a stub must be a stub". This is an honest use of the gate in this context.

**3. Gate tools:**
- **`check_data_cards.py`:** The frontmatter parser is explicitly minimal but robust for its stated constraints. It handles comments and list/scalar fields correctly. It would fail on values with unquoted colons, but this is a reasonable limitation for a tool with no external dependencies.
- **`check_notebooks.py`:** The design is excellent. Coupling the CI tool to the test-file ledger via `ast` is robust, safe, and avoids a brittle dependency on file content or import behavior.
- **`verify.py` & `ci.yml`:** The `verify.py` harness is well-designed. However, the `ci.yml` workflow duplicates its logic instead of calling it directly, creating a needless and risky redundancy. See Finding #3.

**4. Governance coherence:**

- **Data Cards:** Both `DC-usda-nass-quickstats.md` and `DC-kalro-kilifi-crops.md` are valid against the schema enforced by `check_data_cards.py`. There are no inconsistencies between the cards and the code/docs that cite them.
